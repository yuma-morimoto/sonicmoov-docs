# MBO最終報告：予定時間を20%以上超過したタスクを半期で10%以内抑える

## 1. 目標概要

### 背景（As-Is）
- 前回までは「タスク予定時間の95%以内で完了」としていたが、超過タスクに対しての具体的な目標と分析は行えなかった
- 特に修正対応や追加実装などのイレギュラーなタスクでは見積精度が低く、どのようなケースで時間が膨らむのかを把握しきれていなかった
- そのため、どのタスクがどの程度超過しているかに焦点を当てた管理へと移行する必要がある

### 目指す姿（To-Be）
- タスクごとの時間超過を明確に可視化し、大きく超過するタスクを減らす
- 修正や追加実装など、時間超過が発生しやすいタスクの傾向を把握し、見積の精度を改善する

## 3. 実施内容と成果

### 3.1 Backlogでの記録運用
- **実施内容：**
  - Backlog上で全タスクの予定時間・実績時間を記録する
  - 修正タスクについては予定時間を自分で設定する形式で運用
  - 記録内容：タスク名・予定時間・実績時間・超過理由
  - 実績時間は30分単位で記録
- **課題：**
  - 時間管理の漏れにより一部タスクで記録できなかったものがある

### 3.2 超過タスクの分析・対策レポートの作成
- **実施内容：**
  - 半期の超過タスクを集計・分類し、超過要因の分析と今後の対策をまとめた分析レポートを作成した
- **成果：**
	- [超過タスク 分析・対策レポート](https://docs.google.com/spreadsheets/d/1h5IUUP1jjrHSQmHvo4bh5P4m0BzRgXXc/edit?usp=sharing&ouid=103324529737370114241&rtpof=true&sd=true)

### 3.3 成果：プロジェクト全体での予実

- プロジェクト全体で見た場合、計測上では実績時間の合計は予定時間の約62%で完了した。
- 記録漏れがあるため誤差はあるが、それを加味しても予定時間の100%以内に収まっている。
- 超過タスクに焦点を当てると未達が目立つ結果となったが、プロジェクト全体では予定よりも早くタスクをこなせることができた。

## 4. 得られた知見・学び

### 4.1 超過の傾向把握

- 機能開発は見積時に十分な時間を確保していたため超過は少なかった
- 実装後の仕様変更対応やバグ修正での超過が多く、超過タスクの大半がバグ修正
- バグ修正は単一の原因ではなく、複合的な要因が重なって超過した

### 4.2 超過の主な原因

| 原因           | 内容                                            |
| ------------ | --------------------------------------------- |
| 修正対応の往復      | 一度の修正で直しきれず、レビュー後に追加不具合が発覚して何往復も発生した          |
| 動作確認コストの過小見積 | フォームバリデーションなど確認パターンが多い箇所で、動作確認に想定以上の時間がかかった   |
| テストコードの不足    | 自動テストがないため修正の正しさを手動確認に頼らざるを得ず、修正・確認が長くなった     |
| バグ原因調査の長期化   | 表面的な症状から原因を特定するまでに時間を要した                      |
| バグ修正中の仕様変更   | 修正対応中に仕様変更が入り、対応範囲が広がった                       |
| 仕様自体のバグ      | 実装してみて初めて、コードではなく仕様そのものに問題があることが判明した          |
| 既存実装の複雑性     | 既存コードが複雑なため、原因調査・修正・リファクタリングそれぞれに想定以上の時間がかかった |
| バックエンドとの認識齟齬 | APIなどの仕様が固まり切っていない状態で開発が進み、後から仕様変更・修正対応が発生した  |

---

## 5. 今後のアクション

| 対策               | 目的                                   | 対応する原因                  |
| ---------------- | ------------------------------------ | ----------------------- |
| 事前に仕様のすり合わせを行う   | 認識齟齬を事前に解消し、実装中の仕様変更対応を防ぐ            | バックエンドとの認識齟齬、バグ修正中の仕様変更 |
| 着手前に完了条件を洗い出す    | 修正・確認すべきパターンを網羅し、後から追加対応が発生するのを防ぐ    | 修正の往復、動作確認コストの過小見積      |
| バグ対応前に調査タスクを切り出す | 原因調査・影響範囲の把握を見積から分離し、実態に即した再見積を行う    | バグ原因調査の長期化、既存実装の複雑性     |
| 見積チェックシートを作成する   | 見積を漏れなく実施するための仕組みとして、見積項目をチェックリスト化する | 見積漏れ全般                  |
