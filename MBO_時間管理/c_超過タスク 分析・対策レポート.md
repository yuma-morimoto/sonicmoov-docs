## 0. 概要（エグゼクティブサマリー）

| 項目          | 値               |
| ----------- | --------------- |
| 対象期間        | 2025/10〜2026/03 |
| タスク総数       | 62件             |
| 20%以上超過タスク数 | 12件             |
| 超過率         | 19.35%          |
| 目標達成状況      | 未達（目標10%以内）     |

**主な超過要因（上位3点）**

1. 既存実装の複雑性・影響範囲の読み違い（修正タスクで頻発）
2. 動作確認コストの過小見積（複数回修正が必要になるケース）
3. フレームワーク起因の予期しない挙動（Nuxtキャッシュ等）

**来期に向けた主なアクション（上位3点）**

1. 修正・追加実装タスクは「調査タスク」を分離して見積
2. 類似タスクの実績を見積時に参照する
3. 20%超過が発生したら即時振り返りを記録する

## 1. 目的と評価指標の確認

### 目的

本レポートは「予定時間を20%以上超過したタスクを半期で全体の10%以内に抑える」というMBO目標に対して、超過の実態を分析し、次期の改善につなげることを目的とする。

### 評価指標の定義

| 指標 | 定義 |
|------|------|
| 予定時間 | タスク着手前にBacklogに設定した見積時間 |
| 実績時間 | 実際にかかった作業時間（30分単位で記録） |
| 超過率 | （実績 − 予定）÷ 予定 × 100 |
| 20%以上超過タスク | 超過率が20%を上回るタスク |

## 2. 全体集計（月別推移）

| 月 | 総タスク数 | 20%以上超過タスク数 | 超過率 |
|----|-----------|-------------------|--------|
| 10月 | 3件 | 0件 | 0% |
| 11月 | 6件 | 0件 | 0% |
| 12月 | 33件 | 7件 | 21% |
| 1月 | 12件 | 2件 | 16% |
| 2月 | 9件 | 3件 | 33% |
| **合計** | **62件** | **12件** | **19.35%** |

### 傾向考察

- **10〜11月は超過ゼロ**: タスク数が少なく、比較的シンプルな作業が中心だった可能性がある
- **12月に集中（7件/33件）**: タスク量が急増しており、件数の多さに比例して超過が増えた。見積の精度を保てなかった可能性
- **2月が最高超過率（33%）**: タスク数は減っているが超過率は悪化。月末に向けた対応追加や複雑な修正が集中した可能性

## 4. 超過要因の分類・分析

### 超過の傾向

- 機能開発は見積時に十分な時間を確保していたため超過は少なかった
- 実装後の仕様変更対応やバグ修正での超過が多く、超過タスクの大半がバグ修正
- バグ修正は単一の原因ではなく、複合的な要因が重なって超過した

### 要因カテゴリ別集計

| 要因カテゴリ | 件数 | 割合 | 代表例 |
|-------------|------|------|--------|
| 既存実装の複雑性・影響範囲の読み違い | 2件 | 29% | SMV-2408, SMV-2392 |
| 動作確認コストの過小見積 | 2件 | 29% | SMV-2441, SMV-2396 |
| フレームワーク起因の予期しない挙動 | 1件 | 14% | SMV-2488（Nuxtキャッシュ） |
| 楽観的見積（原因不明・記録なし） | 2件 | 29% | SMV-2373, SMV-2439 |


### 種別別傾向

| 種別 | 超過タスク数 | 超過タスク内の割合 |
|------|-------------|------------------|
| バグ修正 | 6件 | 86% |
| 機能実装 | 1件 | 14% |

- **超過はほぼバグ修正に集中**している。修正タスクは既存コードへの影響範囲が読みにくく、見積の難易度が高い
- 機能実装タスクは予実が概ね安定している（SMV-2373の+60%が唯一の超過）
- 最大超過率400%（SMV-2408, SMV-2441）もバグ修正タスク

### 超過の主な原因

| 原因           | 内容                                            |
| ------------ | --------------------------------------------- |
| 修正対応の往復      | 一度の修正で直しきれず、レビュー後に追加不具合が発覚して何往復も発生した          |
| 動作確認コストの過小見積 | フォーム・状態管理など確認パターンが多い箇所で、動作確認に想定以上の時間がかかった     |
| テストコードの不足    | 自動テストがないため修正の正しさを手動確認に頼らざるを得ず、修正・確認のループが長くなった |
| バグ原因調査の長期化   | 表面的な症状から原因を特定するまでに時間を要した                      |
| バグ修正中の仕様変更   | 修正対応中に仕様変更が入り、対応範囲が広がった                       |
| 仕様自体のバグ      | 実装してみて初めて、コードではなく仕様そのものに問題があることが判明した          |
| 既存実装の複雑性     | 既存コードが複雑なため、原因調査・修正・リファクタリングそれぞれに想定以上の時間がかかった |
| バックエンドとの認識齟齬 | APIなどの仕様が固まり切っていない状態で開発が進み、後から仕様変更・修正対応が発生した  |

## 5. 代表ケースの深掘り（ここはいらない消す）

### Case 1: SMV-2408（+300%）

| 項目 | 内容 |
|------|------|
| 種別 | バグ修正 |
| 予定 | 0.5h |
| 実績 | 2h |
| 超過率 | +300% |

1. **タスク概要**: 既存実装の拡張を伴うバグ修正
2. **見積時の前提**: 小さな修正で済むと判断し0.5hで見積
3. **実際に起きたこと**: 既存実装の構造が複雑で、修正の影響範囲が広かった
4. **見積がズレた本質的な原因**: 着手前に既存コードを十分に読まずに見積した
5. **本来どうすべきだったか**: 修正箇所の既存実装を確認してから見積る。不確実性が高い場合は調査タスクを分離する

---

### Case 2: SMV-2441（+300%）

| 項目 | 内容 |
|------|------|
| 種別 | バグ修正 |
| 予定 | 0.5h |
| 実績 | 2h |
| 超過率 | +300% |

1. **タスク概要**: 仮申込再提出画面で、駐車車両に「バイク」を選択した際にバイク排気量が選択できないバグの修正
2. **見積時の前提**: 排気量フィールドの表示制御を直すだけのシンプルな修正と想定し、0.5hで見積
3. **実際に起きたこと**:
   - 初回修正・PR（#267）をマージ後、レビューで2件の追加不具合が発覚
     - 「自動車」選択時にバイク排気量が送信されてしまう問題
     - 「駐車車両 & バイク排気量」を同時に再提出依頼した場合の実装漏れ
   - PR#274・PR#276 と計3回マージが必要になった
   - 最終的に動作確認まで含めて合計2hに到達
4. **見積がズレた本質的な原因**: 再提出依頼のパターン（駐車車両のみ／排気量のみ／両方）を網羅的に把握せず、一つのケースだけ修正すれば済むと判断した。仕様の組み合わせ複雑性を見落としていた
5. **本来どうすべきだったか**: 再提出依頼の対象項目の組み合わせパターンをチケット着手前に洗い出し、修正範囲を明確にしてから見積る。フォームの状態管理を伴う修正は「修正 + 全パターンの動作確認」をセットで見積に含める

---

### Case 3: SMV-2488（+50%）

| 項目 | 内容 |
|------|------|
| 種別 | バグ修正 |
| 予定 | 4h |
| 実績 | 6h |
| 超過率 | +50% |

1. **タスク概要**: Nuxtキャッシュが原因のバグ修正
2. **見積時の前提**: 通常のバグとして4hで見積
3. **実際に起きたこと**: 原因がNuxtフレームワークのキャッシュ挙動にあり、調査に想定外の時間がかかった
4. **見積がズレた本質的な原因**: フレームワーク起因の問題は原因特定の難易度が高く、予測が難しい
5. **本来どうすべきだったか**: 再現条件が絞り込めていない段階では調査時間を多めに見積る。フレームワーク周辺の問題は「わからない時間」込みで見積る意識を持つ

## 6. 半期中の改善施策と効果

### 実施した改善

| 施策 | 内容 |
|------|------|
| Backlogでの記録運用開始 | 全タスクの予定・実績時間をBacklogに記録する体制を構築 |
| 30分単位での実績計上 | 実績時間を細かく計上し、記録精度を高めた |
| 修正タスクの予定時間を自己設定 | 修正タスクも予定時間を自分で設定する運用に統一 |

### 改善の効果

- 記録・可視化体制は構築できた（小目標「毎月記録・洗い出し」は100%達成）
- 月別推移では12月の21%から1月16%に改善傾向が見られた

### 効果が出なかった点・残課題

- **超過率は目標10%に対して19.35%で未達**（達成率51%）
- 2月は33%と悪化。記録体制の構築が見積精度の向上に直結しなかった
- 記録漏れが一部発生し、実態と若干の乖離がある可能性

| ---------------- | ------------------------------------ | ----------------------- |
| 事前に仕様のすり合わせを行う   | 認識齟齬を事前に解消し、実装中の仕様変更対応を防ぐ            | バックエンドとの認識齟齬、バグ修正中の仕様変更 |

| 見積チェックシートを作成する   | 見積を漏れなく実施するための仕組みとして、見積項目をチェックリスト化する | 見積漏れ全般                  |
