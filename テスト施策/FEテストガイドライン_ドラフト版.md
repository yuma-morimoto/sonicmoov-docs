## 0. はじめに

フロントエンドのテスト自動化の導入と運用について体系的にまとめたマニュアルです。
このマニュアルに従ってテスト自動化の導入・運用をしてください。

## 1. 適用範囲
- 担当案件のフロントエンド領域すべて

## 2. テスト計画
### 2-1. 概要
テスト計画は設計段階から着手する

**メリット**：設計段階にテスト計画を立てることで、要件や仕様の漏れに早く気づきやすくなったり、実装中にテストを書きやすいといった利点がある。

### 2-2. 流れ
1.テスト視点で要件定義をチェックする
2.テスト計画は暗黙知にせず文書化する
3.チームに共有できる状態を整える

### 2-3. テスト方針
- この案件でテストをどの程度書くべきか
- 自動テストでどこまで品質を担保するか
- 手動確認に残す範囲はどこか
- 工数とリスクを踏まえて、どのテストを優先するか

### 2-5. テスト観点（どこをテストするか）

#### 2-5-1. 3つのテスト観点

##### 1. 計算・判定・状態管理
入力に対して、正しい計算結果や判定結果が得られることを確認する。  

**対象例：**  
- 金額、日付、件数などの計算
- バリデーション
- 変換処理
- 条件分岐
- 複雑な状態遷移
- 権限や表示可否の判定

特に、境界値、異常値、条件分岐が多い処理を優先する。

##### 2. 外部連携・フロント境界
外部からデータを受け取る、または外部へデータを渡す処理が期待どおりに行われることを確認する。  
  
**対象例：**
- バックエンドAPI、外部API  
- Local Storage、Session Storage  
- Cookie
- URL、クエリパラメータ、ルーティング  
- その他のブラウザAPI
  
通信成功時だけでなく、通信失敗、データなし、不正なデータなども検討する。

##### 3. ユーザー操作と画面上の振る舞い
ユーザーの操作やアプリケーションの状態に応じて、必要な情報や操作結果が画面に表れることを確認する。  
  
**対象例：**
- 入力、クリック、選択、送信  
- バリデーションメッセージ  
- ローディング、空状態、エラー状態  
- ボタンの有効・無効  
- モーダル、アコーディオン、画面遷移  
- APIレスポンスに応じた表示  
  
内部の関数が呼ばれたかではなく、ユーザーから確認できる結果をテストする。

#### 2-5-2. 各対象で確認する状態  
  
2-5-1.「3つのテスト観点」において、以下の状態を必要に応じて確認する。
  
- 正常な入力・操作  
- 境界値  
- 不正な入力  
- データが存在しない状態  
- 通信中
- 通信失敗
- 権限がない状態  
- 操作の重複や連続実行

### 2-6. テスト優先度
- **優先度が高い**
	- 手動テストを何度も繰り返すと想定される箇所
	- バグ発生時の影響度が大きい箇所
- **優先度が低い（テストの保守性と合わせて検討する）**
	- テスト価値が低い箇所
	- 手動テストを何度も行わない箇所
	- 仕様が決定していない又はすぐに変更される可能性のある箇所


## 3.テスト実施

- 純粋関数の境界値
- 静的コンポーネントのテスト（テキスト比較など）は1ファイル1つのみとする
	- 静的なテストは価値が薄いので書く必要はないが、テストすべき理由があればその限りではない。
	- excludeしてはならない、今後拡張された場合にカバレッジが下がらないため。

### 3-1. ウェブアプリ

- ユニット / 統合 / E2E の役割分担
- どのレイヤーに何を書くか
- どの責務はどのレベルで担保するか
- 良いテスト / 悪いテスト
- テストコードを書く標準手順

### 3-2. マークアップ

- 静的検証、アクセシビリティ、表示崩れ検知、ビジュアル差分、必要最小限のE2E
- DOMの細部固定を避ける条件
- 何を自動化し、何を手動確認に残すか
- 良いテスト / 悪いテスト
- 実装・修正時の判断フロー


## 4. 運用・改善

### 4-1. Codecovを活用する

#### 4-1-1. PR品質ゲートの設定
- 例えばテストが75%以上でなければPRをマージできないようにする
- `codecov.yml`に`patch`を定義（targetを75%に）
	- （詳細：「5-3.`codecov.yml`の設定例」）
- GitHub側でマージをブロックする設定を行う
	- Require status checks to pass before merging

### 5-2. 実装サンプル

### 5-3. `codecov.yml`の設定例

```yaml
coverage:
  status:
    project:
      default:
        target: auto
        threshold: 1%
        informational: true
    patch:
      default:
        target: auto
        threshold: 1%
        informational: true

comment:
  layout: "reach,diff,flags,files"
  behavior: default
  require_changes: false

ignore:
  - "**/node_modules/**"
  - "**/dist/**"
  - "**/cdk/**"
  - "**/cdk.out/**"
  - "**/*.test.ts"
  - "**/*.spec.ts"
  - "**/base/**"
  - "**/coverage/**"
```
