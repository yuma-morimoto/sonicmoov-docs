# MEASURE-145 【検証】テスト範囲・テストレイヤー判断基準の標準化

- Backlog: [MEASURE-145](https://sonicmoov.backlog.jp/view/MEASURE-145)
- 状態: 未対応
- 担当者: 小田崎裕美子
- 期間: 2026-06-10〜2026-07-31（着手が9月からだから7月末では無理なのでは？）

## 課題

- テスト範囲・テストレイヤーの判断基準・テスト充足具合が案件ごとにばらついている。
- 実装者のテスト意識のばらつきや属人化により、どのレイヤーで何を担保するかの判断が揃っていない。

## 理想
- どこまでテストするか、どのレイヤーを使うか実装者間で共通認識が取れていて、テストコードもムラがない状態

## 検証したい仮説

- フォーム、スキーマ、画面遷移などの機能単位でテストレイヤー判断基準を作れば、案件ごとの判断差を減らせる。
- テスト対象とテストレイヤーの判断をガイドライン化することで、案件ごとの属人的な判断を減らせる。

以下をテストルールとして共通認識を持つことで、実装者・レビュー者間で、どのレイヤーで何を担保するかの認識合わせがしやすくなるかどうかを検証する。

### テスト観点

| テスト観点           | 第一候補                  | 必要に応じて使うレイヤー       | 主な判断基準                       |
| --------------- | --------------------- | ------------------ | ---------------------------- |
| 計算・判定・状態管理      | Unit                  | Component / E2E    | DOMや通信なしで結果を確認できるか           |
| 外部連携・フロント境界     | Component・Integration | Unit / E2E         | 境界をモックして十分確認できるか、実環境との接続が必要か |
| ユーザー操作と画面上の振る舞い | Component・Integration | E2E / VRT / Manual | 1画面内で完結するか、画面横断や実ブラウザ環境が必要か  |

### テスト範囲・テストレイヤー

| 対象                 | 標準レイヤー                |
| ------------------ | --------------------- |
| 計算・変換・判定・スキーマ      | Unit                  |
| ユーザー操作と画面内の振る舞い    | Component・Integration |
| APIやStorageを含む画面動作 | Component・Integration |
| 複数画面を横断する重要シナリオ    | E2E                   |
| 見た目の意図しない変化        | VRT                   |
| 自動判定が困難な品質         | Manual                |

## アクション

- [ ] ガイドラインに入れる / 入れないを判断する。

## 仮説の有効さを判断する基準

- [ ] テスト対象・テストレイヤーを判断する手間が減った
- [ ] 自動テスト対象や手動確認対象の漏れに気づけた
- [ ] 実装者・レビュー者間で、どのレイヤーで何を担保するかの認識合わせがしやすくなった。
- [ ] 判断理由を説明しやすくなり、属人的な経験だけに頼らずに判断できた。
- [ ] 別案件でも同じ判断基準を使えそうな項目がある。

### ガイドラインに含める内容

- どの機能をテスト対象にするかの判断基準
- Unit / Component / E2E / VRT / Manualの使い分け
- フォーム、スキーマ、画面遷移、API連携などの具体例
- Must / Should / Mayの分類
- 判断に迷うケースと、その理由
