## 1. 目的

- **現状（As-Is）**
	- テストが十分に書けていないが、なぜ不足しているのか整理されていない
	- テストの書きどころや優先順位が感覚や経験に依存している
	- どういったテストが価値を生んでいるのか、効果が可視化されていない
	- テストに関する改善が場当たり的・単発で終わっている

- **あるべき姿（To-Be）**
	- 可能な限りテストを書き、カバレッジ目標へ近づける（本質は品質の向上なので見失わない）
	- テスト実装範囲が明文化されている
	- テスト技法が統一されている
	- テスト導入のガイドラインが存在する

- **なぜ問題なのか（背景・ビジネス影響）**
	- テスト投資の妥当性を説明できず、場当たり的な実装になっている
	- プロジェクトごとに暗黙知化し、形式知にならない

## 4. 施策内容：具体的な実行計画

- **施策の全体像:**
	- **体系化フェーズ：**
		- **Ph1. 現状把握**
			- 現状のテスト自動化の実施の現状を把握する
			- テストがなぜ、どこで、どのように不足、または偏っているのかを具体的に把握し、現状のテスト状況を理解する
		- **Ph2. 改善仮説**
			- 様々なプロジェクトにおいて、どこに、どのようにテスト手法やテスト技術を採用することで、合理的なテストが書けるのか、あらゆるテストの選択肢を取捨選択し、検証するための仮説を立てる
		- **Ph3. 改善仮説検証**
			- 改善仮説を実プロジェクトに適用し、再現性と効果のある体系として落とし込み、テスト標準化の土台とする
		- **標準化フェーズ：**
			- プロジェクトに対し、いつ・どのようなテストを・どのような技法やツールを使って行うかや、責任範囲の明確化、テスト運用ルールなどを標準化する

- **実行手順（概要）:**
	- **体系化フェーズ：**
		- **現状把握**
			- 現状テスト資産の棚卸し
				- テストコード記述量やカバレッジなどの定量情報を取得できる範囲で収集
			- 現状のテストの偏りを可視化
				- ページ別・レイヤー別にどこにテストコードが集中しているか、どこが足りないかを明確にする
			- テストが書かれない・偏る要因の整理
				- 技術や知識、工数の観点からテストが書かれない原因を整理する
		- **改善仮説の立案（考えられる改善策）**
			- テスト観点・テスト手法の選択肢洗い出し
			- 不足している観点に対して、考えうるテスト手法を洗い出す
			- ユニットとして運用できるか、技術的に実現可能かを整理する
			- 改善仮説を立てる
			- 洗い出した各手法について、どの問題を解決できるか、なぜ現状より合理的なのか、どの条件下で効果が出るのか仮説を立てる
			- 改善につながらない手法は捨てる
			- 仮説を検証するための実プロジェクトを選定する
		- **テスト実施・効果の分析**
			- 実プロジェクトへ適用する
			- 立てた仮説をもとに実プロジェクトへ適用する
			- 適用の結果、仮説が正しいかを分析
			- 有効だった考え方・判断基準を体系的に整理
			- 仮説が正しくなかったものは捨てる
			- 標準化施策への引き渡し
			- テスト標準化で使える素材（判断基準・事例・注意点）を整理
	- **標準化フェーズ：**
		- 標準化対象の整理
		- 体系化で得られたノウハウ・知見を、どこまで標準化すべきか見極める
		- 標準化の対象を分類する（Must, Should, Mayで分類）
		- 標準化に向かないものを明確にする（技術変化が激しい、プロジェクト特性依存が強い、試行錯誤段階のもの）
		- テスト導入ガイドラインを作成する

## 6. 予算・リソース：ROIの考え方

- **必要予算:**
	- 合計金額：450,000円（100h）
	- 内訳：
		- 体系化工数：270,000円（60h）（MTGや調査、検証）
		- 標準化工数：180,000円（40h）（MTGや資料作成）

- **必要なリソース（人・物）:**
	- 体系化フェーズの仮説検証に使用する実際の案件（1〜2件）
	- 上記案件のメンバー1名ほど（施策の協力者）

- **投資対効果 (ROI):**
	- 予測ROI（1年間）：132%
	- 算出根拠:
		- 総リターン594,000円 / 投資450,000円 x 100％ ≒ 132%
	- **総リターンの式：**
		1. 削減されるバグ対応 = テストを導入する案件数 x バグ対応の推定工数 x 30%削減
		2. 削減される動作確認 = テストを導入する案件数 x 動作確認の推定工数 x 30%削減 ※
	- ※ 補足：
		- 動作確認工数は、テストコードが仕様確認を兼ねることで、影響範囲確認・手動確認の多くが不要になることを前提とし、削減率を30%と仮定している
	- **総リターンの算出（1年間）：**
		1. 削減されるバグ対応：10案件 x 5人日 x 30%削減 = 540,000円（120h）削減
		2. 削減される動作確認：10案件 x 0.5人日（30タスクx10分）x 30% = 54,000円（16h）削減
