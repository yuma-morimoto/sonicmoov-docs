# コーポレートサイトのテスト計画

## 0. 前提

- 対象プロジェクト: `/Users/l_0026/Documents/corporate-site-microcms`
- 対象サイト: ソニックムーブ コーポレートサイト
- 技術構成: Next.js 16 / React 19 / microCMS / Amplify Hosting
- 既存テスト設定: Vitest は導入済み。ただし現時点ではテストファイルは未配置
- 本計画は `テスト施策/FEテストマニュアル.md` の「テスト計画」方針をもとに、コーポレートサイト向けに具体化する

## 1. テスト方針

このサイトでは、すべてのページを同じ粒度で自動テストするのではなく、壊れたときの影響度が高い導線、変更頻度が高い箇所、外部サービス連携を優先してテストする。

特に以下を重視する。

- microCMS から取得した情報が、一覧・詳細・タグ絞り込み・下書きプレビューで正しく表示されること
- 問い合わせ、会社案内ダウンロード、採用、実績、お知らせなど、ユーザー行動につながる導線が失われないこと
- ヘッダー、フッター、ページタイトルなど、全ページ共通のレイアウトが崩れないこと
- 現行サイトから移植した見た目やレスポンシブ表示が、主要画面幅で大きく崩れないこと
- SEO / OGP / canonical など、コーポレートサイトとして必要なメタ情報が欠落しないこと

## 2. 品質基準

開発前または対象機能の着手前に、ディレクターとエンジニアで以下を確認する。

- どのページ・導線がリリース前に必ず確認すべき重要シナリオか
- 現行サイトとの差分をどこまで許容するか
- 自動テストで担保する範囲と、手動確認に残す範囲
- microCMS の実データ、モックデータ、プレビュー用データのどれを使って確認するか
- 外部フォームや外部リンクなど、自動テストで深追いしない範囲の確認方法

## 4. テスト種別と役割

### 4-1. 静的検証

目的は、実装ミスや型の不整合を早い段階で検出すること。

- `npm run lint`
- `npm run build`
- microCMS のスキーマ変更後は `npm run gen-api-types` を実行し、不要な型変更が混ざっていないか確認する

### 4-2. ユニットテスト

対象は、画面表示から切り出せる純粋なロジックや、条件分岐のある小さな関数に限定する。

候補:

- 日付表示の変換
- タグ絞り込み条件
- URL / クエリ文字列の生成
- microCMS 取得結果の整形
- Draft Mode の対象 endpoint 判定

### 4-3. コンポーネントテスト

ユーザー操作や画面上の振る舞いを確認する、ブラックボックステストであるコンポーネントテストを中心に実施する。

候補:

- `ContentList`
  - 初期状態では全件が表示される
  - タグを選択すると対象コンテンツだけが表示される
  - `All` を選択すると絞り込みが解除される
- ヘッダー
  - SP メニューを開閉できる
  - サブメニューを開閉できる
  - PC 表示では主要ナビゲーションが表示される
- フッター
  - ページトップボタンが押せる
  - 外部リンクに `target="_blank"` と `rel="noopener noreferrer"` が付与される
- 採用職種一覧
  - カテゴリごとに募集職種が表示される
  - 募集職種リンク押下時の GTM イベントが発火する

### 4-4. E2Eテスト

E2Eテストは実装・保守コストが高いため、リグレッション防止の価値が高い主要導線に絞る。こちらも、ユーザー操作や画面上の振る舞いを確認するブラックボックステストとして扱う。

初期導入では Playwright を想定する。

優先シナリオ:

- お知らせ一覧から詳細ページへ遷移できる
- お知らせ一覧でタグ絞り込みができる
- 実績一覧から詳細ページへ遷移できる
- 実績一覧でタグ絞り込みができる
- お問い合わせページでフォーム iframe が表示される
- 会社案内ダウンロードページでフォーム iframe が表示される
- SP 幅でヘッダーメニューを開き、主要ページへ遷移できる
- フッターの主要リンクから各ページへ遷移できる

### 4-5. ビジュアル・レスポンシブ確認

現行サイトからの移植差分が多いため、表示確認は重要度が高い。ただし、初期段階では自動のビジュアル差分よりも、重要ページに絞ったスクリーンショット確認を優先する。

確認幅:

- SP: 375px
- Tablet: 768px
- PC: 1440px

優先ページ:

- `/news`
- `/news/[id]`
- `/works`
- `/works/[id]`
- `/contact`
- `/download`
- `/company`
- `/jobs`

確認観点:

- ヘッダー・フッターが重ならない
- ページタイトルと本文の余白が崩れていない
- 画像の縦横比が崩れていない
- タグ、ボタン、リンクの折り返しが不自然でない
- iframe の高さが不足していない
- 現行サイト移植の修正メモにある差分が解消されている

### 4-6. 手動確認に残す範囲

以下は自動テストで完全に担保しにくいため、リリース前の手動確認に残す。

- Leadpad フォーム内の入力・送信完了までの動作
- microCMS 管理画面からの入稿・公開フロー
- 下書きプレビュー URL の実運用確認
- 外部サイト側のページ内容
- 現行サイトとの細かな視覚差分
- 実機でのスクロール、タップ、アニメーションの自然さ

## 5. リスクベースの優先順位

### 高リスク

- microCMS のデータ取得に失敗し、一覧・詳細ページが表示できない
- タグ絞り込みや詳細リンクが機能せず、コンテンツにたどり着けない
- お問い合わせ・ダウンロードフォームが表示されない
- SP メニューが開けず、スマートフォンで主要ページへ遷移できない
- リリース後に SEO / OGP / canonical が欠落する

### 中リスク

- 現行サイトとの見た目の差分が大きい
- 採用職種のカテゴリ表示や外部リンクが崩れる
- 画像サイズや本文 HTML により詳細ページのレイアウトが崩れる
- ページトップボタンや外部リンクの補助導線が機能しない

### 低リスク

- 固定文言中心のページで、軽微な余白差分がある
- 影響範囲の狭い装飾やホバー表現の差分がある

## 7. 実装順序

1. `npm run lint` と `npm run build` をリリース前の必須確認にする
2. `ContentList` のコンポーネントテストを追加する
3. ヘッダーの SP メニュー・サブメニューのコンポーネントテストを追加する
4. `/news` と `/works` の主要 E2E を追加する
5. `/contact` と `/download` の iframe 表示 E2E を追加する
6. 主要ページのスクリーンショット確認を追加する
7. microCMS の下書きプレビュー確認を必要に応じて追加する

## 9. 運用ルール

- 新しいページを追加した場合は、まずリスクを「高・中・低」に分類し、高リスクの導線だけでもテスト対象に追加する
- microCMS のスキーマを変更した場合は、型生成、一覧表示、詳細表示、プレビュー表示を確認する
- 共通レイアウトを変更した場合は、最低でも `/news`、`/works`、`/contact`、`/download` の SP / PC 表示を確認する
- E2E テストは増やしすぎず、ユーザーが実際に通る重要導線を優先する
- 自動テストで確認しない項目は、リリース前チェックに手動確認として明記する
