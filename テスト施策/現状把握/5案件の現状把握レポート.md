## 全体サマリ

5案件とも、何らかの自動テスト資産とCI連携は存在している。ただし、テストの厚さ・レイヤー構成・運用成熟度には大きな差がある。

全体として共通しているのは、ロジック・スキーマ・ユーティリティなど、比較的切り出しやすい領域からテストが導入されている点である。一方で、画面全体、ページ遷移、外部サービス連携、実機・WebView・CMSとの結合、フォーム送信などは手動確認に残りやすい。

特に標準化の検討価値が高いのは、以下の3点である。

- テスト対象の判断基準: ロジック、UI表示分岐、重要フロー、外部連携をどのレイヤーで担保するか
- 手動確認に残す範囲の明文化: 実機、WebView、LINE/LIFF、CMS、外部決済、外部フォームなど
- CI・PR運用の標準化: lint / typecheck / unit / component / coverage / E2E のどこまでをPR時に必須にするか

## チェックリスト別の整理

### 1. 現在あるテスト資産

#### 共通する部分

- 5案件すべてに自動テスト資産が存在する。
- 5案件すべてでCIまたはローカルフックにより、少なくとも一部のテストを実行する仕組みがある。
- Codecovやカバレッジレポートを使い、テストの有無や偏りを可視化する動きがある。
- 後付け導入の案件が多く、最初から全レイヤーを揃えるよりも、テストしやすい領域から段階的に追加している。

#### 異なる部分

| 観点         | 厚い案件                          | 薄い案件                   |
| ---------- | ----------------------------- | ---------------------- |
| テスト件数      | ナビパーク、イースト                    | ほほえみごはん、コーポレート、TRP     |
| E2E        | ナビパーク、イースト                    | TRP、ほほえみごはん、コーポレート     |
| コンポーネントテスト | イースト、コーポレート、一部ナビパーク           | TRP、ほほえみごはん            |
| カバレッジ運用    | ナビパーク、TRP、イースト、ほほえみごはん、コーポレート | ただし計測対象・CI対象ブランチには差がある |

#### 特徴

- イーストはファイル命名とVitest projectsにより、レイヤーがかなり明確に分かれている。
- ナビパークはテスト資産の量と種類が最も多く、Unit / E2E / MSW / Storybook / Chromatic まで揃っている。
- コーポレートはNext.js App Routerの制約に合わせて、node環境とjsdom環境を分けている点が特徴。
- TRPはBrowserテストの設定があるが、まだテストファイルがなく、導入直前の状態に見える。
- ほほえみごはんはAstro構成のため、ページ・レイアウトよりも `scripts/` のDOM操作ロジックにテストが寄っている。

### 2. どのレイヤーにどんなテストがあるか

#### 共通する部分

- Unitテストはどの案件にも存在する。
- Unitテストは、バリデーション、日付・年齢計算、金額計算、文字列処理、API Route、CMSクライアント、DOM操作ロジックなど、比較的独立させやすい処理に集中している。
- 統合テストは5案件とも明示的なレイヤーとしては薄い。
- E2Eはある案件でも対象が限定的で、すべての主要フローを網羅しているわけではない。

#### 異なる部分

- ナビパークは、Unitでロジックを押さえつつ、決済・駐車場詳細・法人仮申込・再提出などをE2Eで一部押さえている。
- イーストは、Unit / Browser / E2E の3層が比較的きれいに分かれている。
- TRPはUnitのみが実運用され、BrowserとE2Eは未活用。
- ほほえみごはんはUnitのみで、Astroコンポーネントやページのテストはない。
- コーポレートは、lib/API Routeをnode環境、Client Componentをjsdom環境で分けているが、ページ統合やE2Eはない。

#### 標準化の示唆

テストレイヤーの使い分けは、以下のように整理できそうである。

| 対象 | 推奨レイヤー |
| --- | --- |
| 入力値の形式チェック、相関バリデーション、金額・日付計算 | Unit |
| API Route、外部APIクライアント、CMS取得処理の分岐 | Unit |
| UIの表示分岐、ユーザー操作、共通コンポーネント | Component / Browser Component |
| 入力から完了までの主要フロー、決済・申込・OAuthなど失敗影響が大きい導線 | E2E |
| 実機、WebView、LINE/LIFF、GPS、外部決済、実CMS、外部フォーム連携 | 手動確認項目として明文化。可能な範囲だけモック化して自動化 |

#### 共通する部分

- 重要なロジックや仕様分岐がある箇所にはテストが入りやすい。
- ページ全体や画面遷移は、どの案件でも手薄になりやすい。
- 共通部品のうち、表示分岐やインタラクションがあるものはテスト対象になっている場合がある。

#### 案件ごとの特徴

| 案件 | テストがある主な機能 | 手薄な主な機能 |
| --- | --- | --- |
| ナビパーク | 月極申込・再提出・時間貸決済・駐車場詳細・バリデーション・utils | マイページ、カード管理、会員登録/ログイン単独フロー、通知、FAQ、問い合わせ、外部連携 |
| イースト | Suica連携、OAuth、エラーページ、友だち判定、連携ステータス取得 | LIFF実環境、実Suica API、統合レイヤー |
| TRP | 申し込み/見積もりスキーマ、年齢計算、日時、文字列、useCounter | フォーム全体、子ども情報入力、多言語、郵便番号API、LIFF/LINE連携 |
| ほほえみごはん | もっと見る、リアクション、GTMイベント、デバッグログ | アコーディオン、カルーセル、モーダル、検索、スクロール、全ページレンダリング |

#### 議論ポイント

- 「重要フロー」を案件ごとにどう定義するか。
- ページ単位のテストを増やすべき案件と、ロジック・部品単位に留めるべき案件をどう分けるか。
- 共通コンポーネントは、静的表示だけなら手動確認でよいのか、全ページ影響を理由に最低限のテストを持つべきか。

### 4. テストの偏り

#### 共通する部分

- テストはロジック・スキーマ・utilsに偏りやすい。
- 画面全体、ページ遷移、外部サービスとの結合、実機依存の動作は薄くなりやすい。
- 後付け導入の場合、直近で変更された機能や不具合が出た機能にテストが集中しやすい。

#### 案件ごとの偏り

| 案件 | 厚い領域 | 手薄な領域 |
| --- | --- | --- |
| ナビパーク | バリデーション、月極、決済、純粋関数、E2E主要フロー | ページ単位、マイページ、カード管理、WebView/外部連携 |
| イースト | OAuth/Suica連携、エラーページ、Browserテスト | lib/api除外領域、統合テスト、実環境連携 |
| TRP | Zodスキーマ、年齢・日時・文字列utils | フォームコンテナ、動的UI、多言語、外部API |
| ほほえみごはん | useSingle、useReadMore、GTM検証 | UIインタラクション全般、Astroページ、レスポンシブ |
| コーポレート | microCMS、Draft API、共通・専用コンポーネント | page.tsx、SEO、フォーム、CMS結合、E2E |

#### 標準化の示唆

偏り自体は悪いことではない。むしろ、限られた工数では影響が大きく、壊れやすく、テストしやすい場所から入るのが現実的である。

ただし、以下の偏りは標準化やチェックリストで補正した方がよい。

- 変更頻度が高いのにテストがない領域
- 手動確認が多いのに確認項目が明文化されていない領域
- 共通部品なのに、壊れた場合の影響範囲が大きい領域
- CIではテストしているが、対象ブランチや実行条件が限定されていて日常運用に乗っていない領域

### 5. 手動確認の実態

#### 共通する部分

5案件とも、自動化しきれない領域が存在する。特に以下は共通して手動確認に残りやすい。

- 実機やWebViewでしか確認しづらい挙動
- LINE/LIFF、OAuth、Cognito、Flutter bridgeなどの外部認証・アプリ連携
- microCMS、Suica API、郵便番号API、外部決済、外部フォームなどの外部サービス連携
- ページ全体の表示、レスポンシブ、ホバー、アニメーション、レイアウト崩れ
- フォームの入力から完了までの一連の流れ
- SEO/OGP、外部リンク、実データとの結合

#### 異なる部分

- ナビパークはモバイルアプリWebView、Cognito、Google Maps、決済など、実行環境依存が特に強い。
- イーストとTRPはLINE/LIFFの実環境確認が重い。
- ほほえみごはんはUIインタラクションとページ表示・レスポンシブ確認が中心。
- コーポレートはmicroCMS、Draft Mode、ISR、問い合わせ・ダウンロードフォーム、SEO/OGPが中心。

#### 標準化の示唆

自動化できない領域を「未対応」として放置するのではなく、手動確認項目として標準化する価値が高い。

特にPRテンプレートやテスト観点表に、以下のような項目を持たせると議論しやすい。

- 自動テストで確認した範囲
- 手動確認した範囲
- 実機・外部サービス・本番相当環境でしか確認できない範囲
- 今回確認しない範囲と、その理由

### 6. 不具合や変更が起きやすい領域

#### 共通する部分

不具合や変更が起きやすい領域は、おおむね以下に集約できる。

- 入力項目が多く、条件分岐が複雑なフォーム
- 金額、決済、申込、契約、連携状態など、事業影響が大きい処理
- 外部サービスとの境界
- 多言語、翻訳、メッセージ、表示条件
- 全ページに影響する共通部品
- CMSスキーマやAPI仕様の変更影響を受ける処理

#### 案件ごとの特徴

| 案件 | 変更・不具合リスクが高い領域 |
| --- | --- |
| ナビパーク | 月極申込・再提出・更新、決済、認証、駐車場検索/詳細、WebView/外部連携 |
| イースト | OAuth state、連携ステータスキャッシュ、エラーハンドリング、友だち判定 |
| TRP | 申し込みフォーム、子ども情報入力、多言語、郵便番号連動、フォーム完了フロー |
| ほほえみごはん | リアクション、GTM/GA、もっと見る、カルーセル、全体JSエントリ |
| コーポレート | ContentListタグフィルタ、Draft API、microCMS取得、Header/Footer、採用・お知らせ・実績 |

#### 標準化の示唆

「変更頻度が高い」「影響範囲が大きい」「手動確認が重い」の3条件が重なる箇所を、優先的なテスト追加候補にするとよい。

案件ごとに技術は違っても、優先順位の判断軸は共通化できる。

### 7. なぜ現状のテスト実態になっているか

#### 共通する背景

- 既存案件に後からテストを導入しているため、一気に網羅するのではなく段階的に増やしている。
- スキーマ、utils、API Route、CMSクライアントなどはテストしやすく、最初の導入対象になりやすい。
- UI、ページ、外部連携、実機確認はセットアップやモックのコストが高く、手動確認に残りやすい。
- CIやカバレッジ基盤はあっても、実際にどこまで品質ゲートとして使うかは案件ごとに差がある。

#### 案件別の背景

- ナビパークは、複雑な条件分岐を純粋関数・スキーマに切り出す方針があり、Unitテストが厚い。E2Eは重要フローに後追いで追加されている。
- イーストは、アプリ規模が比較的小さく、Suica連携・OAuthというコア機能が明確なため、レイヤー分離された自動テストを組みやすい。
- TRPは、フォームロジックはZodスキーマでテストしやすい一方、Conform、React Router hooks、LIFF、LINE連携が絡む画面全体はテストコストが高い。
- ほほえみごはんは、AstroコンポーネントやCSSアニメーションの自動テストが難しく、TypeScriptのDOM操作ロジック中心になっている。
- コーポレートは、Next.js App RouterのServer ComponentやCMS結合のテストが難しく、lib/API Route/Client Componentから導入している。

## 標準化の検討価値がある部分

### 1. テスト対象の判断基準

案件横断で使えそうな判断基準は以下である。

| 判断軸 | テスト導入の優先度が高い条件 |
| --- | --- |
| 影響度 | 決済、申込、認証、CMS表示、フォーム送信など、失敗時の影響が大きい |
| 変更頻度 | 直近で変更が多い、今後も仕様変更が見込まれる |
| 複雑さ | 条件分岐、入力項目、状態管理、外部連携が多い |
| 手動確認コスト | 毎回の確認に時間がかかる、確認漏れが起きやすい |
| 再利用性 | 共通部品、共通ロジック、複数画面で使われる |

### 2. レイヤー分離の標準パターン

以下のようなレイヤー分離を、FEテストマニュアルや案件導入時のチェックリストに落とし込めそうである。

| レイヤー | 主な対象 | 代表的な案件例 |
| --- | --- | --- |
| Unit | スキーマ、utils、APIクライアント、計算、バリデーション | ナビパーク、TRP、コーポレート |
| Component / Browser | 表示分岐、ユーザー操作、共通UI、フォーム部品 | イースト、コーポレート、ナビパークの一部 |
| E2E | 重要なユーザージャーニー、申込、決済、OAuth、完了フロー | ナビパーク、イースト |
| VRT / Storybook | 見た目の差分、共通部品、デザイン影響が大きいUI | ナビパーク |
| Manual | 実機、WebView、LINE/LIFF、CMS実データ、外部決済、SEO/OGP | 全案件 |

### 3. CI・PR運用

以下は標準化候補として扱える。

- PR時に最低限 `lint` / `typecheck` / `unit test` を実行する。
- E2Eや重いテストは、対象ブランチ・手動実行・夜間実行など、案件に合わせて実行条件を分ける。
- Codecovは絶対的な目標値ではなく、テストの偏りや未検証領域を見つける材料として使う。
- PRテンプレートに「自動テストで確認した範囲」と「手動確認した範囲」を分けて書く。
- テストコマンド・CI実行条件・カバレッジ対象を README / AGENTS / TEST.md などに明記する。

### 4. 手動確認項目の標準化

自動化できない領域は、手動確認リストとして標準化する価値がある。

| 手動確認カテゴリ | 対象例 |
| --- | --- |
| 実機・WebView | ナビパーク、イースト、TRP |
| 外部認証・連携 | Cognito、OAuth、LINE/LIFF、Suica API |
| 外部サービス | microCMS、郵便番号API、決済、GAS、フォームサービス |
| 表示確認 | レスポンシブ、ホバー、アニメーション、カルーセル、モーダル |
| コンテンツ | SEO/OGP、外部リンク、CMSデータ、翻訳文言 |
| リリース影響 | ISR、プレビュー、stg/prod差分、環境変数差分 |

## 第2回MTGで議論したいこと

### 1. 標準化する対象をどこに置くか

現状把握から見ると、全案件に一律で「E2Eを必ず入れる」とするよりも、以下を標準化対象にする方が現実的である。

- テスト対象を選ぶ判断基準
- Unit / Component / E2E / Manual の使い分け
- 手動確認に残す項目の書き方
- CI・PRで最低限確認する範囲
- 後付け導入時の最初の一歩

### 2. Webアプリとマークアップ・コーポレートサイトを分けて考えるか

ナビパーク、イースト、TRPのようなアプリ系は、フォーム・認証・決済・連携などの重要フローが中心になる。

ほほえみごはんやコーポレートのようなサイト系は、ページ表示、CMS、SEO、タグ、レスポンシブ、共通レイアウトの確認が中心になる。

そのため、共通の原則は持ちつつ、チェックリストは以下の2系統に分けると使いやすそうである。

- アプリ系: フォーム、認証、決済、外部連携、状態管理、E2E
- サイト系: ページ表示、CMS、SEO/OGP、タグ、レスポンシブ、共通レイアウト

### 3. 次に検証する改善仮説

現状把握から考えると、次に検証する改善仮説の候補は以下である。

| 改善仮説 | 期待効果 | 検証しやすい対象 |
| --- | --- | --- |
| 変更頻度が高いフォームにComponentまたはE2Eを追加すると、手動確認工数を減らせる | フォーム確認の抜け漏れ削減 | TRP、ナビパーク |
| 手動確認項目をPRテンプレート化すると、確認範囲の属人化を減らせる | 確認漏れ・認識ずれの削減 | TRP、ほほえみごはん、コーポレート |
| レイヤー別の命名規則・設定を標準化すると、後付け導入がしやすくなる | テスト追加の心理的コスト削減 | イースト、コーポレート |
| CMS・API・外部連携の境界をUnitでモック化すると、仕様変更時のデグレを見つけやすくなる | CMS/API周辺のデグレ削減 | コーポレート、TRP |
| UIインタラクションの共通パターンをテスト化すると、サイト系の手動確認を減らせる | カルーセル、モーダル、検索などの確認工数削減 | ほほえみごはん |

## まとめ

5案件の現状把握から、テスト導入は「テストを書くかどうか」よりも、「どのリスクをどの確認方法で担保するか」を案件ごとに決める段階にあることが見える。

共通化すべきなのは、特定のツールやテスト件数ではなく、以下のような判断と運用である。

- 影響度・変更頻度・複雑さ・手動確認コストからテスト対象を選ぶ
- Unit / Component / E2E / Manual の役割を分ける
- 自動化できない領域を手動確認項目として明文化する
- CI・PRでどこまで確認するかを案件ごとに決め、ドキュメントに残す
- カバレッジは目標値ではなく、偏りと次の投資対象を見つける材料として使う

第2回MTGでは、この整理をもとに「標準化する項目」と「次に検証する改善仮説」を絞り込む。
