## 3. 実施内容と成果

### 3.1 行った取り組み

#### E2E テスト環境構築
- **実施内容**：
  - Playwright を導入し、Page Object Model を採用して保守しやすい基盤を構築
  - ローカル実行環境と GitHub Actions の workflow を作成
- **成果物**：
  - zenn記事：[E2EのPOMを関数ベースで書いてみた](https://zenn.dev/sonicmoov/articles/ec18ed5659907c)
  - GitHub Actions での Playwright 実行は workflow 作成まで完了。実行時エラーの調査・修正は今後の課題。

#### 法人対応のE2Eテスト作成
- **実施内容：**
  - 主要画面のユーザーフローを対象に E2E テストを実装
  - **テスト対象画面：** 駐車場詳細 / 仮申込 / 再提出
  - 要素特定は `role`・`label`・`text` などアクセシビリティ情報を使用
    （css や `data-testid` は使用しない方針）

### 3.2 コンポーネントテストの見送りについて
- **判断内容：** チーム合意により E2E テストを優先し、本格実装を見送り
- **判断理由：**
  - ユーザーフロー品質を優先する方針をチームで合意していた
  - Playwright によるコンポーネントテストの有効性調査を並行実施したため着手が後ろ倒しになった
- **影響：**
  - 本格導入には至らなかったが、フォームバリデーションへの適用が有効で
    保守も容易との知見が得られ、今後の課題が明確になった

## 4. 未達項目と要因

### 4.1 未達項目

- **法人対応のアプリ側のE2Eテストが書けた**：全体の20%達成（主要3画面程度まで実装）

### 4.2 要因分析

**主な要因：**
- 工数と優先度の兼ね合いから影響度の高い画面を優先したため、
  実装が主要3画面程度に留まり全体網羅には至らなかった

## 5. 得られた知見・学び

- POM + fixture の組み合わせで E2E テストの保守性が向上する
- MSW モックのファクトリパターン化でモック管理コストを削減できる
- Playwright コンポーネントテストはフォームバリデーションに有効
- バックエンド接続 E2E テストはモック設計が Flaky 対策の鍵

## 6. 課題と今後のアクション

- 単体テストカバレッジの維持・向上に向けたテスト追加
-  E2Eテストケースの拡充
- GitHub Actions 上での Playwright 失敗原因を調査し、CI で安定実行できるよう修正
- フォームバリデーションへの Playwright コンポーネントテスト適用方針を整理
