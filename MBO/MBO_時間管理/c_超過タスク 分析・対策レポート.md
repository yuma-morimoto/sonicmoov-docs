		- 見積時に不確定要素（バグの複雑度、既存実装の把握、影響範囲、機能の複雑度）の大きさも加味する
		- 見積チェックシートを作って見積もりの漏れを減らす

## 3. 超過要因の傾向・分析

### 月ごとの超過の傾向
- **10〜11月**：機能実装が主な作業だったのとタスク数が少なかったため超過はなかった。
- **12〜1月**：プロジェクトはテスト期間に入りフィードバックやバグ修正のタスクが増えた。バグ修正では予定時間を溢れてしまうことが多かった。
- **2月**：タスク数は減っているが超過率は悪化した。リリースに向けた対応の追加や端末依存の複雑な修正が集中した。

### 全体での超過の傾向

- 機能実装は見積時に十分な時間を確保していたため超過は少なかった
- 実装後の仕様変更対応やバグ修正での超過が多く、超過タスクの大半がバグ修正
- バグ修正は単一の原因ではなく、いくつかの要因が重なって超過した

### 超過の主な原因

| 原因           | 内容                                            |
| ------------ | --------------------------------------------- |
| バグ原因調査の長期化   | 表面的な現象から原因を特定するまでに時間がかかった                     |
| 修正対応の往復      | 一度の修正で直しきれず、レビュー後に追加不具合が発覚して何往復もした            |
| 動作確認コストの過小見積 | フォーム・状態管理など確認パターンが多い箇所で、動作確認に想定以上の時間がかかった     |
| テストコードの不足    | 自動テストがないため修正の正しさを手動確認に頼らざるを得ず、修正・確認のループが長くなった |
| バグ修正中の仕様変更   | 修正対応中に仕様変更が入り、対応範囲が広がった                       |
| 仕様自体のバグ      | 実装してみて初めて、コードではなく仕様そのものに問題があることが判明した          |
| 既存実装の複雑性     | 既存コードが複雑なため、原因調査・修正・リファクタリングそれぞれに想定以上の時間がかかった |
| バックエンドとの認識齟齬 | APIなどの仕様が固まり切っていない状態で開発が進み、後から仕様変更・修正対応が発生した  |

---

## 4. 今後の対策

| 対策               | 目的                                   | 対応する原因                  |
| ---------------- | ------------------------------------ | ----------------------- |
| バグ対応前に調査タスクを切り出す | 原因調査・影響範囲の把握を見積から分離し、実態に即した再見積を行う    | バグ原因調査の長期化、既存実装の複雑性     |
| 見積チェックシートを作成する   | 見積を漏れなく実施するための仕組みとして、見積項目をチェックリスト化する | 見積漏れ全般                  |
| 事前に仕様のすり合わせを行う   | 認識齟齬を事前に解消し、実装中の仕様変更対応を防ぐ            | バックエンドとの認識齟齬、バグ修正中の仕様変更 |
| 着手前に完了条件を洗い出す    | 修正・確認すべきパターンを網羅し、後から追加対応が発生するのを防ぐ    | 修正の往復、動作確認コストの過小見積      |
