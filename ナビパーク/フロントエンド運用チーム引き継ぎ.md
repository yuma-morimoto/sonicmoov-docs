## プロジェクト概要

- ナビパーク駐車場の時間貸決済・月極契約をアプリ上から行えるサービス
- 時間貸決済：駐車場の決済をアプリ上からできる機能（クレジットカードや PayPay など現金以外の決済にも対応）
- 月極契約：駐車場の月極契約をアプリ上から申請できる機能
- 公式ページ： https://www.navipark1.com/navi-app/
- フロントエンドが担当する領域は **アプリ** と **管理画面** の2つ
	- **アプリ**：Nuxt 3 + Vite + TypeScript の SPA。Flutter アプリの WebView 上で表示される
	- **管理画面**：Laravel の MVC 上で Nuxt が SPA として表示される。Blade ファイルのみフロント側で管理が必要

## 4. システム概要

`navipark_frontend` は Nuxt 3 + Vite + TypeScript で構築された SPA で、Flutter アプリの WebView 上で表示される。

主な特徴:

- `ssr: false` の SPA 構成
- ビルドは `nuxi generate`
- Flutter との連携に `webview-javascript-bridge` を利用
- 主要機能は、駐車場検索、時間貸決済、定期利用券、月極契約、お知らせ、FAQ、クーポン、夢なび連携

## 5. 環境一覧

| 環境 | URL | 用途 | 備考 |
| --- | --- | --- | --- |
| local | `http://localhost:3333` | ローカル開発 | `.env.local` を使用 |
| dev | `https://dev.app.navipark1.com` | 開発・テスト | Flutter 側と接続した確認が必要 |
| stg | `https://stg.app.navipark1.com` | クライアント確認 | Flutter 側と接続した確認が必要 |
| prod | `https://app.navipark1.com` | 本番 | Flutter 側と接続した確認が必要 |

補足:

- `dev / stg / prod` は API 認証の都合で、**通常ブラウザだけでは完全な確認ができない**
- QR 遷移や OAuth / 決済完了復帰では `static.app.navipark1.com` 系の static サイトを経由する

## 6. ブランチ運用・CI/CD

### ブランチ

環境とブランチの対応は以下。

| 環境 | 監視ブランチ |
| --- | --- |
| dev | `develop` |
| stg | `staging` |
| prod | `master` |

`navipark_frontend/README.md` では、上記ブランチへのマージで各環境へ自動デプロイされる前提になっている。

### ローカルの保護

`navipark_frontend` では Husky により、`develop` / `staging` / `master` への直接 commit / push を防止している。



### AWS ビルド / デプロイ


引き継ぎ時に共有すること:

- AWS コンソールで確認するサービス
  - CodeBuild
  - CodeDeploy
  - CodePipeline
- デプロイ成否を確認する Slack チャンネル
- 失敗時に最初に見るログ
  - GitHub Actions
  - CodeBuild ログ
  - Sentry

`navipark_ops_ecs/ops/README.md` は AWS 運用全般の共通手順として参考になるが、**フロントエンドの実際のビルド内容は `navipark_frontend` 側を優先して参照する**。


### Sentry

主な確認箇所:

- `src/plugins/sentry.client.ts`
- `buildspec.yml`

運用上のポイント:

- `local` / `test` 以外で Sentry が有効
- build 時に sourcemap をアップロードしている
- 一部の既知エラーは `ignoreErrors` で除外している
  - Flutter を介さずブラウザで見たときのエラー
  - Google Maps 内部エラー
  - 一部のバリデーション / 決済キャンセル系エラー

引き継ぎ時に共有すること:

- Sentry のプロジェクト
- アラート通知先
- 本番障害時の初動確認手順
- 除外中の既知エラーの考え方

### 障害切り分けの観点

障害時はまず以下を切り分ける。

- ブラウザ単体で再現するか
- Flutter WebView 上でのみ再現するか
- QR / OAuth / 決済復帰など static サイト経由でのみ再現するか
- 環境依存か（dev / stg / prod）

特に以下はフロント単独で完結しない。

- QR 遷移
- SNS ログイン
- 決済完了後の戻り

## 13. 引き継ぎ時に口頭で補足したいこと

以下は文書だけでなく、口頭でも補足したい。

- dev / stg / prod は Flutter 側と接続しないと正しく確認できない箇所がある
- QR / OAuth / 決済復帰はフロント単独では切り分けられない
- Sentry の既知除外エラーがあるため、件数だけでなく中身を見る必要がある
- 月極契約まわりの自動テストはまだ十分ではない
- お知らせ / FAQ / メンテナンス画面は、運用設定変更がそのまま表示品質に影響する

- 直近の未解決 Backlog を把握している
